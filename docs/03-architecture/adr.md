# Architecture Decision Records (ADR)

## ADR-001: Выбор архитектурного паттерна PCMEF

**Контекст:** Необходимо выбрать архитектуру для информационной системы проката автомобилей в рамках курсового проекта по дисциплине «Программная инженерия».

**Решение:** Использовать архитектурный паттерн PCMEF (Presentation → Control → Mediator → Entity → Foundation) с разделением на 5 слоёв.

**Обоснование:**
- Паттерн обеспечивает строгую Layer Isolation — каждый слой зависит только от нижележащего
- Соответствует требованиям методических указаний
- Обеспечивает testability и maintainability

**Последствия:**
- Presentation слой реализован на Android (Kotlin + Compose)
- Control слой — Spring REST-контроллеры
- Mediator — сервисы бизнес-логики с паттернами State и Strategy
- Entity — JPA-сущности с доменной логикой
- Foundation — JPA-репозитории и Room DAO

## ADR-002: Выбор мобильной платформы Android (Kotlin)

**Контекст:** Необходимо выбрать платформу для мобильного клиента.

**Решение:** Использовать Android Native с Kotlin и Jetpack Compose.

**Обоснование:**
- Соответствует траектории «Мобильная разработка»
- Kotlin — современный язык со статической типизацией
- Jetpack Compose — современный declarative UI toolkit от Google

**Последствия:**
- Используется MVVM + StateFlow для управления состоянием
- Retrofit для сетевых запросов
- Room для локального кэша
- Material 3 Design System

## ADR-003: Использование State-паттерна для бронирований

**Контекст:** Жизненный цикл бронирования включает статусы PENDING → CONFIRMED → ACTIVE → CANCELLED/COMPLETED. Необходимо обеспечить корректные переходы между статусами и блокировать невалидные операции.

**Решение:** Реализовать паттерн State (ReservationState → PendingState, ConfirmedState, ActiveState, CancelledState).

**Обоснование:**
- Каждый статус инкапсулирует своё поведение
- Невозможны невалидные переходы (каждый State бросает IllegalStateException для неподдерживаемых операций)
- Легко добавлять новые статусы

**Последствия:** Reservation содержит @Transient метод getState(), который по enum ReservationStatus возвращает нужную реализацию State.

## ADR-004: Использование Strategy для ценообразования

**Контекст:** Стоимость аренды зависит от дней недели (будни/выходные). Необходимо подставлять алгоритм расчёта динамически.

**Решение:** Реализовать паттерн Strategy (PricingStrategy → StandardPricingStrategy, WeekendPricingStrategy).

**Обоснование:**
- Легко добавлять новые тарифные планы
- Алгоритмы изолированы друг от друга
- Принцип Open/Closed соблюдён

**Последствия:** В `ReservationServiceImpl.selectStrategy()` выбирается стратегия на основе проверки `WeekendPricingStrategy.hasWeekend()`.

## ADR-005: Использование Room для офлайн-режима

**Контекст:** Приложение должно работать без интернета, отображая кэшированные данные каталога автомобилей.

**Решение:** Использовать Room (библиотека Android Jetpack).

**Обоснование:**
- Room — официальная рекомендация Google для локального хранения
- Предоставляет type-safe DAO и корутины
- Не требует ContentProvider

**Последствия:** CatalogViewModel при ошибке сети загружает данные из Room-кэша и отображает индикатор «Офлайн-режим». Кэш обновляется при успешной загрузке полного списка.

## ADR-006: Использование EncryptedSharedPreferences для JWT

**Контекст:** JWT-токен содержит конфиденциальные данные и должен храниться безопасно.

**Решение:** Использовать EncryptedSharedPreferences из AndroidX Security.

**Обоснование:**
- Шифрование AES256-GCM на уровне файла SharedPreferences
- Автоматическое управление ключами через Android KeyStore
- Минимальное влияние на производительность

**Последствия:** TokenStorage хранит JWT в зашифрованном виде. JwtInterceptor автоматически подставляет Bearer-токен в каждый запрос.
