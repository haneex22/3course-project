# 10 Развёртывание и администрирование

## 10.1 Цель этапа

Целью данного этапа является подготовка серверной части информационной системы проката автомобилей к эксплуатации, настройка базы данных PostgreSQL, запуск мобильного Android-приложения и проверка взаимодействия компонентов системы посредством REST API.

---

## 10.2 Требования к программному окружению


| Компонент      | Версия          | Назначение                             |
| -------------- | --------------- | -------------------------------------- |
| Java           | 21              | Выполнение серверной части             |
| Maven          | 3.9+            | Сборка Spring Boot проекта             |
| PostgreSQL     | 15+             | Хранение данных                        |
| Android Studio | Hedgehog и выше | Разработка и запуск мобильного клиента |
| Android SDK    | API 26+         | Выполнение мобильного приложения       |
| Docker Desktop | 24+             | Контейнеризация серверной части        |
| Git            | 2.x+            | Получение исходного кода проекта       |


---

## 10.3 Подготовка окружения

Перед запуском системы необходимо выполнить установку требуемого программного обеспечения:

1. Java Development Kit (JDK 21);
2. Apache Maven;
3. PostgreSQL;
4. Android Studio;
5. Docker Desktop;
6. Git.

После установки рекомендуется проверить корректность установки следующими командами:

```bash
java -version
mvn -version
git --version
docker --version
psql --version

```

---

## 10.4 Настройка базы данных

Перед первым запуском серверной части необходимо создать базу данных PostgreSQL.

Пример создания базы данных:

```sql
CREATE DATABASE carrentaldb;

```

После создания базы данных необходимо указать параметры подключения в файле конфигурации Spring Boot (`application.yml`):

- адрес сервера PostgreSQL;
- порт подключения;
- имя базы данных;
- имя пользователя;
- пароль пользователя.

При первом запуске приложения Hibernate автоматически создаёт необходимые таблицы в базе данных на основании JPA-сущностей.

---

## 10.5 Сборка серверной части

Для сборки серверной части необходимо перейти в каталог backend и выполнить команду:

```bash
mvn clean package

```

После успешной сборки в каталоге `target` формируется исполняемый JAR-файл приложения.

---

## 10.6 Запуск серверной части

Запуск приложения может быть выполнен одним из двух способов.

### Запуск через Spring Boot

```bash
mvn spring-boot:run

```

### Запуск JAR-файла

```bash
java -jar target/*.jar

```

После запуска сервер становится доступен по адресу:

```text
http://localhost:8080

```

---

## 10.7 Запуск мобильного приложения

Для запуска Android-клиента необходимо:

1. открыть проект в Android Studio;
2. выполнить синхронизацию Gradle;
3. подключить физическое устройство либо запустить Android Emulator;
4. нажать кнопку **Run**.

После запуска приложение автоматически выполняет взаимодействие с серверной частью посредством REST API.

---

## 10.8 Проверка работоспособности

После запуска необходимо проверить корректность функционирования системы.

Рекомендуется выполнить следующие проверки:

- регистрация нового пользователя;
- авторизация пользователя;
- получение списка автомобилей;
- просмотр карточки автомобиля;
- создание бронирования;
- вход под учетной записью администратора;
- просмотр административного раздела.

При успешном выполнении указанных операций можно считать, что серверная и клиентская части функционируют корректно.

---

## 10.9 Администрирование

Администрирование системы осуществляется пользователем с ролью **ADMIN**.

Для демонстрации проекта используется следующая тестовая учетная запись:

**Логин:** `admin@carrent.ru`

**Пароль:** `password123`

Администратор имеет возможность:

- просматривать список автомобилей;
- добавлять новые автомобили;
- редактировать информацию об автомобилях;
- удалять автомобили;
- изменять статус автомобиля;
- просматривать список пользователей;
- просматривать бронирования.

Для проверки функциональности менеджера используется учетная запись:

**Логин:** `manager@carrent.ru`

**Пароль:** `password123`

Менеджер имеет возможность просматривать информацию о бронированиях, пользователях и текущем состоянии автопарка.

---

## 10.10 Аутентификация и безопасность

Для защиты REST API используется механизм JWT-аутентификации.

После успешного входа сервер формирует JSON Web Token, который сохраняется в мобильном приложении и автоматически добавляется в каждый защищённый HTTP-запрос посредством `JwtInterceptor`.

Доступ к административным функциям предоставляется только пользователям с соответствующей ролью.

---

## 10.11 Резервное копирование базы данных

Для создания резервной копии базы данных PostgreSQL рекомендуется использовать утилиту `pg_dump`.

Пример создания резервной копии:

```bash
pg_dump -U postgres carrentaldb > backup.sql

```

Для восстановления базы данных используется команда:

```bash
psql -U postgres carrentaldb < backup.sql

```

## 10.12 Возможные ошибки


| Проблема                                     | Возможная причина                       | Способ устранения                          |
| -------------------------------------------- | --------------------------------------- | ------------------------------------------ |
| Сервер не запускается                        | PostgreSQL не запущен                   | Проверить работу сервера PostgreSQL        |
| Ошибка подключения к БД                      | Неверные параметры подключения          | Проверить настройки `application.yml`      |
| Ошибка авторизации                           | Неверный логин или пароль               | Проверить учетные данные пользователя      |
| Ошибка JWT                                   | Недействительный или просроченный токен | Выполнить повторную авторизацию            |
| Android-приложение не подключается к серверу | Неверный адрес REST API                 | Проверить URL сервера в настройках клиента |


---

## 10.13 Завершение работы

После завершения работы рекомендуется:

- выполнить выход из учетной записи;
- корректно остановить серверное приложение;
- при использовании Docker остановить контейнеры;
- выполнить резервное копирование базы данных при необходимости.

Корректное завершение работы обеспечивает сохранность данных и предотвращает повреждение базы данных PostgreSQL.
