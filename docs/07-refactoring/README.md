# Этап 7. Рефакторинг

## Цель этапа

Целью этапа являлось повышение качества программного кода, улучшение его сопровождаемости, уменьшение связанности компонентов системы и приведение архитектуры к принципам многослойного приложения.

В процессе разработки был проведён анализ существующего кода, устранены дублирующиеся участки логики, выделены отдельные сервисы бизнес-логики и оптимизировано взаимодействие между слоями приложения.

---

## Выявленные проблемы

В ходе анализа были выявлены следующие недостатки первоначальной реализации:

- дублирование логики обработки бронирований в различных компонентах;
- смешивание бизнес-логики и логики взаимодействия с базой данных;
- повторяющееся преобразование сущностей в DTO;
- высокая связанность отдельных компонентов системы;
- отсутствие единой точки выполнения комплексной операции бронирования автомобиля.

---

## Выполненный рефакторинг

В процессе рефакторинга были выполнены следующие изменения:

- бизнес-логика перенесена в сервисный слой (Mediator);
- доступ к данным полностью сосредоточен в репозиториях (Foundation);
- реализовано разделение ответственности между контроллерами, сервисами и репозиториями;
- выделена фасадная логика оформления бронирования (`BookingFacade`);
- реализованы стратегии расчёта стоимости аренды (`PricingStrategy`);
- реализована модель состояний бронирования (`ReservationState`);
- уменьшено дублирование кода за счёт повторного использования сервисов и DTO.

---

## Использованные шаблоны проектирования

В процессе рефакторинга применены следующие шаблоны проектирования:


| Паттерн    | Назначение                                                                                   |
| ---------- | -------------------------------------------------------------------------------------------- |
| Facade     | Выполнение полного сценария бронирования автомобиля через единый интерфейс (`BookingFacade`) |
| Strategy   | Выбор алгоритма расчёта стоимости аренды в зависимости от условий бронирования               |
| State      | Управление жизненным циклом бронирования и изменением его состояний                          |
| Repository | Абстрагирование работы с базой данных                                                        |
| DTO        | Передача данных между клиентом и сервером без использования JPA-сущностей                    |


---

## Повышение качества кода

После проведения рефакторинга были достигнуты следующие результаты:

- уменьшено количество дублирующегося кода;
- улучшена читаемость проекта;
- повышена модульность компонентов;
- упрощено тестирование бизнес-логики;
- снижена связанность между слоями приложения;
- обеспечено соответствие архитектуре PCMEF.

---

## Проверка результатов

После завершения рефакторинга выполнено повторное тестирование приложения.

Проведена проверка:

- регистрации пользователей;
- авторизации;
- получения каталога автомобилей;
- оформления бронирования;
- изменения статусов автомобилей;
- административных операций;
- работы JWT-аутентификации.

Все ранее разработанные модульные и интеграционные тесты успешно пройдены, что подтверждает корректность внесённых изменений.

---

## Ключевые артефакты этапа

В результате выполнения этапа были получены следующие результаты:

- оптимизированная архитектура серверной части;
- выделенный сервисный слой;
- использование шаблонов Facade, Strategy, State и Repository;
- повторно используемые DTO;
- обновлённые модульные и интеграционные тесты.
