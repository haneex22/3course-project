Цель работы: cпроектировать многослойную архитектуру программной системы на основе паттерна PCMEF с учётом выбранной траектории разработки (Desktop/Web/Mobile).
Краткое описание проекта
Предлагается разработать мобильное приложение, интегрированное с информационной системой прокатной компании. Приложение позволит клиенту искать и бронировать автомобили, проходить идентификацию, оплачивать аренду, получать договор и уведомления, а сотрудникам - управлять автопарком, выдачей, возвратом и аналитикой.
Перечень решаемых задач
Сократить время оформления аренды минимум на 50%.
Повысить долю онлайн-бронирований до 70% и выше.
Снизить количество ошибок в расписании, выдаче и возврате автомобилей на 85-90%.
Увеличить коэффициент загрузки автопарка на 20-25%.
Краткая характеристика проекта
Информационная система обеспечивает сквозную автоматизацию процессов удаленного поиска, краткосрочного бронирования, онлайн-оплаты и последующего контроля выдачи/возврата транспортных средств. Мобильное приложение служит единой точкой взаимодействия клиента с сервисом проката, в то время как серверная часть обеспечивает централизованное выполнение жестких бизнес-правил компании.

2. Выбор и обоснование архитектурного стиля
Обоснование выбора PCMEF 
Разделение ответственности: Слои пользовательского интерфейса полностью изолированы на мобильном клиенте (Presentation), тогда как критическая бизнес-логика и правила тарификации инкапсулированы на сервере (Mediator). 
Обеспечение оффлайн-режима: Мобильная среда характеризуется нестабильным интернет-соединением. Модификация PCMEF за счет локального кэширования (Local Cache) позволяет клиенту просматривать ранее сохраненный каталог машин без доступа к сети. 
Безопасность данных: Чувствительные данные (хэши паролей, финансовые транзакции) обрабатываются исключительно на сервере в закрытом контуре СУБД, не перегружая память телефона. 
Тестируемость: Изоляция слоев позволяет тестировать бизнес-сервисы (Mediator) независимо от особенностей рендеринга Flutter-компонентов на клиенте.

3. Диаграмма пакетов PCMEF (Mobile)
Описание каждого слоя
Presentation (мобильный клиент)
Назначение: Отвечает за визуализацию данных, отрисовку графического интерфейса пользователя (UI) и обработку пользовательских жестов (нажатия, свайпы). Вся логика этого слоя сведена к рендерингу.
Ключевые классы: CatalogScreen, BookingScreen, ProfileScreen
Технологии: Android SDK, Jetpack Compose, XML Layouts, Navigation Component
State Management (мобильный клиент)
Назначение: Служит «мозгом» мобильного клиента. Этот слой отделяет интерфейс от данных. Он принимает события от UI (например, «пользователь нажал кнопку фильтрации»), запрашивает данные из сети или кэша, преобразует их в новое состояние (State) и реактивно отправляет обратно в Presentation слой для перерисовки экрана.
Ключевые классы: CatalogBloc, AuthBloc
Технологии: Android ViewModel, LiveData, StateFlow, Coroutines Flow
API Client (мобильный клиент)
Назначение: Инкапсулирует в себе всю логику межпроцессного взаимодействия с удаленным сервером. Отвечает за создание асинхронных HTTPS-соединений, сериализацию объектов Dart в JSON-строки и десериализацию ответов сервера. Также этот слой автоматически прикрепляет токен безопасности (JWT) к каждому запросу.
Ключевые классы: CarApiClient, AuthInterceptor
Технологии: Retrofit, OkHttp, Moshi/Gson, Kotlin Coroutines
Local Cache (мобильный клиент)
Назначение: обеспечивает автономность (оффлайн-режим) мобильного приложения. Если у клиента пропал интернет, слой State Management обращается не к сети, а к этому слою, чтобы показать сохраненный список машин и историю поиска.
Ключевые классы: LocalCarBox, SearchHistoryCache
Технологии: Room, SQLite, Flow для наблюдения за кэшем
Control (сервер)
Назначение: Входная точка серверной части системы (граничный слой). Его задача — принять входящий HTTP-запрос от мобильного приложения, проверить валидность переданных данных (входной контроль DTO с помощью аннотаций @Valid), проверить права доступа пользователя (Spring Security) и передать вызов на уровень бизнес-логики.
Ключевые классы: CarRestController, BookingRestController
Технологии: Spring Boot, @RestController, @Valid
Mediator (сервер)
Назначение: бизнес-логика системы (правила выдачи, расчёт штрафов, бронирование).
Ключевые классы: CarRentServiceImpl, PaymentServiceImpl
Технологии: Spring Service, @Transactional
Entity (сервер)
Назначение: бизнес-сущности, отображаемые на базу данных.
Ключевые классы: Car, Reservation
Технологии: JPA, Hibernate, @Entity
Foundation (сервер)
Назначение: доступ к данным, CRUD-операции, кастомные запросы.
Ключевые классы: CarRepository, ReservationRepository
Технологии: Spring Data JPA, @Repository

    List<Reservation> findActiveReservationsByClientId(@Param("clientId") Long clientId);
    @Modifying
    @Query("UPDATE Reservation r SET r.status = 'CANCELLED' WHERE r.status = 'PENDING' AND r.createdAt < :timeout")
    void cancelOverdueReservations(@Param("timeout") LocalDateTime timeout);
}

Проверка отсутствия циклических зависимостей
На клиенте: Presentation -> State Management -> API Client -> Local Cache. Обратные ссылки отсутствуют, так как слои кэша и сети изолированы от UI-компонентов и отдают только чистые структуры данных (DTO).  
На сервере: Control -> Mediator -> Foundation -> Database. Сущности (Entity) не содержат ссылок на инфраструктуру и репозитории, выполняя требования чистой направленности графа зависимостей вниз.  
6. Адаптация PCMEF под Mobile-траекторию
Особенности адаптации для мобильной траектории
Разрыв слоя Presentation и Control: Они разделены физической сетью.Presentation живет на смартфонах клиентов, а Control преобразован в REST-контроллеры бэкенда. 
Внедрение State Management: Мобильные операционные системы динамически уничтожают и пересоздают контекст экранов (например, при повороте устройства). Слой управления состоянием защищает бизнес-данные сессии от уничтожения. 
Разделение сущностей (Entities): Сущности на сервере (@Entity JPA) описывают полные реляционные схемы со связями OneToMany. Локальные сущности на клиенте представляют собой плоские, легковесные структуры для сохранения в key-value хранилище мобильного устройства
Сравнительная таблица с классическим PCMEF
Обоснование необходимости адаптации
Физическое разделение – клиент и сервер работают на разных устройствах.
Сетевая нестабильность – нужен оффлайн-режим (Local Cache).
Реактивность UI – требуется State Management для эффективного обновления экранов.
Безопасность – чувствительная логика и данные должны быть на сервере.
Экономия трафика – кэширование уменьшает число запросов.

Альтернатива | Причина отказа
Только локальное приложение (SQLite) | Невозможность синхронизации: Изменения статусов автомобилей (например, бронирование машины одним клиентом) должны мгновенно отражаться у всех пользователей. Хранение данных только на устройствах привело бы к критическим конфликтам и повторным арендам одного авто.
Толстый клиент с прямой БД | Невозможность синхронизации: Изменения статусов автомобилей (например, бронирование машины одним клиентом) должны мгновенно отражаться у всех пользователей. Хранение данных только на устройствах привело бы к критическим конфликтам и повторным арендам одного авто.
