5.1. Паттерн State (Состояние)
Где применяется: В сущности Reservation (Бронирование) для управления ее жизненным циклом.
Зачем: Бронирование в каршеринге проходит через строгий конечный автомат: PENDING (Ожидает оплаты) $\rightarrow$ CONFIRMED (Оплачено) $\rightarrow$ ACTIVE (Авто выдано) $\rightarrow$ COMPLETED (Завершено). Действия, доступные пользователю, зависят от текущего статуса (например, нельзя отменить уже активную аренду).
Улучшение: Паттерн позволяет избежать громоздких конструкций if-else или switch-case внутри сервисов при проверке возможности перехода. Логика каждого состояния изолирована в собственном классе.
// Интерфейс Состояния
public interface ReservationState {
    void confirmPayment(Reservation context);
    void handoverCar(Reservation context);
    void cancel(Reservation context);
}
// Конкретное состояние: Ожидание оплаты
public class PendingState implements ReservationState {
    @Override
    public void confirmPayment(Reservation context) {
        // Логика обработки успешной оплаты
        context.setState(new ConfirmedState());
    }
    @Override
    public void handoverCar(Reservation context) {
        throw new IllegalStateException("Нельзя выдать неоплаченный автомобиль");
    }
    @Override
    public void cancel(Reservation context) {
        context.setState(new CancelledState());
    }
}
// Конкретное состояние: Оплачено и Подтверждено
public class ConfirmedState implements ReservationState {
    @Override
    public void confirmPayment(Reservation context) {
        throw new IllegalStateException("Бронь уже оплачена");
    }
    @Override
    public void handoverCar(Reservation context) {
        // Логика выдачи ключей и авто
        context.setState(new ActiveState());
    }
    @Override
    public void cancel(Reservation context) {
        // Логика возврата средств с удержанием комиссии
        context.setState(new CancelledState());
    }
}
// Контекст (Сущность)
@Entity
public class Reservation {
    @Transient // JPA не сохраняет сам объект состояния, только Enum статус
    private ReservationState currentState;
    public Reservation() {
        this.currentState = new PendingState(); // Начальное состояние
    }
    public void setState(ReservationState state) {
        this.currentState = state;
    }
    public void processPayment() {
        currentState.confirmPayment(this);
    }
    public void startRental() {
        currentState.handoverCar(this);
    }
}
5.2. Паттерн «Strategy» (Стратегия)
Где применяется: В PricingService для динамического расчета стоимости аренды автомобилей.
Зачем: В прокате автомобилей существуют разные правила ценообразования: стандартный тариф, повышенный тариф выходного дня (Dynamic Pricing) или тариф со скидкой для лояльных клиентов (списание бонусов).
Улучшение: Алгоритм расчета цены инкапсулируется в отдельные классы-стратегии. Это позволяет гибко добавлять новые маркетинговые акции, не изменяя базовый код расчета и соблюдая принцип Open/Closed.
// Стратегия (Интерфейс)
public interface PricingStrategy {
    BigDecimal calculatePrice(Car car, long rentalDays);
}
// Конкретная стратегия 1: Базовый тариф
@Component
public class StandardPricingStrategy implements PricingStrategy {
    @Override
    public BigDecimal calculatePrice(Car car, long days) {
        return car.getBaseDailyRate().multiply(BigDecimal.valueOf(days));
    }
}
// Конкретная стратегия 2: Тариф выходного дня (+20% к стоимости)
@Component
public class WeekendPricingStrategy implements PricingStrategy {
    @Override
    public BigDecimal calculatePrice(Car car, long days) {
        BigDecimal basePrice = car.getBaseDailyRate().multiply(BigDecimal.valueOf(days));
        return basePrice.multiply(new BigDecimal("1.20")); // Наценка 20%
    }
}
// Контекст (Сервис), использующий стратегию
@Service
public class ReservationService {
    private PricingStrategy pricingStrategy;
    // Стратегия может меняться динамически в рантайме
    public void setPricingStrategy(PricingStrategy strategy) {
        this.pricingStrategy = strategy;
    }
    public Reservation createReservation(Car car, ClientProfile client, long days) {
        // Делегирование расчета выбранной стратегии
        BigDecimal finalPrice = pricingStrategy.calculatePrice(car, days);
        Reservation res = new Reservation(car, client, finalPrice);
        return res;
    }
}
5.3. Паттерн «Factory Method» (для создания репозиториев)
Где применяется: На границе слоев Control и Mediator в виде BookingFacade или RentalOrchestrator.
Зачем: Процесс успешного бронирования (или выдачи авто) требует сложной оркестрации: нужно проверить доступность авто (CarService), заблокировать средства на карте (PaymentService), сохранить бронь (ReservationService) и отправить push-уведомление клиенту (NotificationService).
Улучшение: Паттерн предоставляет единый, упрощенный интерфейс для REST-контроллера. Контроллеру не нужно знать о существовании множества внутренних подсистем, он просто вызывает один метод Фасада, снижая связность системы (Coupling).
// Фасад
@Service
@RequiredArgsConstructor
public class BookingFacade {
    // Внедрение сложных подсистем бэкенда
    private final CarService carService;
    private final PaymentService paymentService;
    private final ReservationService reservationService;
    private final NotificationService notificationService;
    // Единый метод, скрывающий сложность взаимодействия
    @Transactional
    public ReservationResponseDto processCompleteBooking(UUID clientId, UUID carId, PaymentTokenDto token, DateRange dates) {
        // 1. Проверяем авто и временно бронируем
        carService.verifyAvailabilityAndLock(carId, dates);
        // 2. Рассчитываем стоимость и холдируем средства
        BigDecimal amount = reservationService.calculateCost(carId, dates);
        paymentService.holdDeposit(token, amount);
        // 3. Сохраняем финальную бронь
        Reservation reservation = reservationService.saveConfirmedBooking(clientId, carId, dates, amount);
        // 4. Отправляем асинхронное уведомление
        notificationService.sendPushNotification(clientId, "Ваш автомобиль успешно забронирован!");
        return new ReservationResponseDto(reservation);
    }
}
// REST Контроллер (Слой Control) становится максимально тонким
@RestController
@RequestMapping("/api/v1/bookings")
@RequiredArgsConstructor
public class BookingController {
    private final BookingFacade bookingFacade;
    @PostMapping
    public ResponseEntity<ReservationResponseDto> bookCar(@RequestBody BookingRequest req) {
        // Контроллер работает только с фасадом
        var response = bookingFacade.processCompleteBooking(
            req.getClientId(), req.getCarId(), req.getPaymentToken(), req.getDates()
        );
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
}

Контрольные вопросы
1. Что такое диаграмма последовательности и для чего она используется?
Диаграмма последовательности — это тип диаграмм UML, которая показывает взаимодействие объектов во времени. Она отображает последовательность обмена сообщениями между объектами, временную упорядоченность вызовов методов и жизненные циклы объектов. Используется для детального проектирования сценариев использования, выявления недостающих методов и атрибутов, проверки архитектурных решений (например, соблюдения слоев PCMEF) и документирования сложных бизнес-процессов.
2. Назовите основные элементы диаграммы последовательности
Основными элементами диаграммы последовательности являются: линия жизни (lifeline), которая представляет объект или компонент и обозначается в формате "Имя:Класс"; актор (actor) — внешний пользователь или система; сообщение (message) — вызов метода от одного объекта к другому, обозначается стрелкой →; возврат (reply) — возврат результата, обозначается пунктирной стрелкой - - →; создание объекта — обозначается стрелкой с надписью [new]; уничтожение объекта — обозначается символом X; а также комбинированные фрагменты (combined fragments) — alt, opt, loop — для показа условных и циклических конструкций.
3. Как на диаграмме последовательности показать условное выполнение?
Условное выполнение показывается с помощью комбинированного фрагмента alt (alternative). Синтаксис следующий: сначала пишется ключевое слово "alt", затем условие, после двоеточия — действия, выполняемые при истинности условия. Далее может следовать блок "else" с другим условием, и так далее. Завершается фрагмент ключевым словом "end". Например, в сценарии выдачи книги проверяется наличие штрафов: если у читателя есть штраф, выбрасывается исключение; иначе (else) выполняется выдача книги.
4. Как показать цикл на диаграмме последовательности?
Цикл показывается с помощью комбинированного фрагмента loop. Синтаксис: после ключевого слова "loop" в скобках указывается условие повторения или количество итераций, затем перечисляются сообщения, которые должны повторяться, и в конце ставится "end". Например, можно показать цикл обработки списка книг: loop (for each book in books) — затем выполняется какая-то операция с каждой книгой.
5. В чем разница между синхронным и асинхронным сообщением?
Синхронное сообщение — это вызов метода, при котором отправитель блокируется и ждет ответа от получателя, прежде чем продолжить выполнение. На диаграмме обозначается сплошной стрелкой с закрашенным наконечником (→). Асинхронное сообщение — это вызов, при котором отправитель не ждет ответа и продолжает выполнение сразу после отправки сообщения. На диаграмме обозначается сплошной стрелкой с открытым наконечником (→>). Пример синхронного вызова: UI вызывает метод контроллера и ждет результата. Пример асинхронного: отправка уведомления на email — сервис отправил и пошел дальше, не дожидаясь доставки.
6. Какие методы выявляются при построении диаграмм последовательности?
При построении диаграмм последовательности выявляются следующие методы: методы граничных классов (UI), которые принимают действия пользователя; методы контроллеров, которые обрабатывают запросы от UI; методы сервисов (Mediator слой), которые содержат бизнес-логику; методы репозиториев (Foundation слой) для доступа к данным; а также методы сущностей (Entity), которые реализуют предметно-ориентированную логику, такую как checkAvailability(), isAvailable(), registerUser() и другие.
7. Как диаграммы последовательности связаны с архитектурой PCMEF?
Диаграммы последовательности напрямую отражают архитектуру PCMEF, так как на них явно выделяются слои: Presentation (boundary-классы), Control (control-классы), Mediator (сервисы), Entity (сущности) и Foundation (репозитории, БД). Каждый объект на диаграмме принадлежит одному из этих слоев, а стрелки вызовов должны строго идти сверху вниз: от Presentation к Control, затем к Mediator, потом к Entity и Foundation. Обратные стрелки — это только возвраты результатов. Таким образом, диаграмма последовательности является проверкой соблюдения архитектурных принципов.
8. Почему важно соблюдать направление вызовов сверху вниз (P → C → M → E → F)?
Соблюдение направления вызовов сверху вниз важно по нескольким причинам. Во-первых, это обеспечивает четкое разделение ответственности между слоями — каждый слой знает только о нижележащих слоях. Во-вторых, это предотвращает появление жестких связей (например, когда Presentation напрямую обращается к БД, минуя бизнес-логику). В-третьих, это упрощает тестирование — каждый слой можно тестировать изолированно, подменяя нижележащие слои моками. В-четвертых, это позволяет легко заменять реализации слоев (например, заменить Foundation с JPA на JDBC), не затрагивая вышележащие слои.
9. Как на диаграмме последовательности показать создание нового объекта?
Создание нового объекта показывается стрелкой сообщения с деструктором «new». На диаграмме это выглядит так: объект-создатель отправляет сообщение новому объекту, и в тексте сообщения указывается "new" или имя конструктора. Линия жизни нового объекта начинается в том месте, куда приходит это сообщение. Например: Service -> LoanEnt: new(readerId, bookId, dueDate). После этого у созданного объекта активируется линия жизни, и он может получать дальнейшие сообщения.
10. Какие паттерны GoF наиболее часто используются при детальном проектировании?
Наиболее часто используемые паттерны GoF при детальном проектировании: Strategy — для реализации различных алгоритмов (например, расчет штрафов или цены); Observer — для уведомления об изменениях (например, обновление UI после выдачи книги); Command — для инкапсуляции запросов (например, отмена или повтор операций); Factory Method и Abstract Factory — для создания семейств объектов (например, создание репозиториев для разных БД); Facade — для упрощения интерфейса сложной подсистемы (например, API клиента); Template Method — для определения общего алгоритма с вариациями (например, базовый класс репозитория).
11. Как обеспечить соответствие между диаграммами последовательности и диаграммой классов?
Соответствие между диаграммами последовательности и диаграммой классов обеспечивается следующим образом: во-первых, все объекты, участвующие в последовательностях, должны быть описаны на диаграмме классов. Во-вторых, все методы, вызываемые на диаграммах последовательности, должны присутствовать в соответствующих классах на диаграмме классов с правильными сигнатурами (типы параметров и возвращаемых значений). В-третьих, типы возвращаемых значений должны совпадать. Для проверки соответствия можно использовать инструменты автоматической генерации кода из UML или проводить ревью, где сравниваются обе диаграммы. В ходе лабораторной работы для этого заполняется таблица выявленных методов, а затем эти методы добавляются на диаграмму классов.
12. Какие проблемы могут возникнуть, если не построить диаграммы последовательности до реализации?
Если не построить диаграммы последовательности до реализации, могут возникнуть следующие проблемы: во-первых, нарушение архитектуры PCMEF — разработчики могут начать вызывать методы репозиториев напрямую из контроллеров, минуя сервисный слой. Во-вторых, выявление недостающих методов происходит уже в процессе кодирования, что приводит к частым изменениям уже написанного кода. В-третьих, сложные сценарии с альтернативными потоками и обработкой ошибок могут быть реализованы неполно или с логическими ошибками. В-четвертых, увеличивается время на отладку и рефакторинг. В-пятых, документация не соответствует реализации, что затрудняет поддержку проекта в будущем.
13. Как проверить, что диаграмма последовательности не нарушает архитектурные принципы?
Чтобы проверить, что диаграмма последовательности не нарушает архитектурные принципы, нужно выполнить следующие проверки: во-первых, проследить, чтобы все стрелки вызовов шли строго сверху вниз от Presentation к Control, затем к Mediator, затем к Entity и Foundation — не должно быть вызовов снизу вверх (например, Entity не должен вызывать методы Control). Во-вторых, проверить, что Presentation не обращается напрямую к Mediator или Foundation — только через Control. В-третьих, убедиться, что Control не содержит бизнес-логики — вся бизнес-логика должна быть в Mediator (сервисах) и Entity. В-четвертых, проверить, что Foundation (репозитории) не вызывают Mediator или Control. Если все эти условия соблюдены, диаграмма соответствует принципам PCMEF.
14. Приведите пример, когда применение паттерна Strategy улучшает архитектуру.
Пример из лабораторной работы — расчет штрафов за просрочку возврата книг. Без паттерна Strategy пришлось бы писать условные операторы внутри метода расчета штрафа, например: if (libraryType.equals("STANDARD")) { ... } else if (libraryType.equals("PROGRESSIVE")) { ... } else if (libraryType.equals("STUDENT")) { ... }. Это нарушает принцип открытости/закрытости — при добавлении нового типа расчета нужно изменять существующий код. С паттерном Strategy создается интерфейс FineCalculationStrategy, и каждая стратегия реализует свой алгоритм расчета. Сервис LoanService получает нужную стратегию через конструктор или сеттер. Теперь при добавлении новой стратегии (например, "льготный период для пенсионеров") не нужно изменять код LoanService — достаточно создать новый класс, реализующий интерфейс. Архитектура становится гибкой и легко расширяемой.
15. Как показать на диаграмме последовательности взаимодействие с внешним сервисом (например, платежным шлюзом)?
Взаимодействие с внешним сервисом на диаграмме последовательности показывается с помощью отдельного участника (lifeline), который обычно обозначается стереотипом «external» или «participant». Название такого участника отражает внешний сервис, например «PaymentService» или «NotificationService». Вызовы к внешнему сервису изображаются обычными стрелками сообщений. Важно отметить, что внешний сервис находится на том же уровне, что и Foundation слой, или даже ниже — он не должен вызывать внутренние компоненты системы. 
Например, в сценарии выдачи книги сервис LoanService отправляет сообщение NotificationService.sendNotification(readerId, message), получает ответ OK и продолжает работу. Также можно показать обработку ошибок от внешнего сервиса с помощью фрагмента alt, если сервис может вернуть ошибку.
