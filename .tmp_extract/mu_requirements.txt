 ЧАСТЬ 1. ОРГАНИЗАЦИОННО-МЕТОДИЧЕСКАЯ 
 
 1.1. Цель и задачи курсового проекта 
Цель курсового проекта : Закрепление и углубление теоретических 
знаний, полученных в курсе «Программная инженерия», а также 
формирование практических навыков проектирования, разработки и 
документирования программного обеспечения на всех этапах жизненного 
цикла от бизнес -анализа до развертывания готового продукта с 
использованием архитектурного паттерна PCMEF (Presentation -Control-
Mediator-Entity-Foundation). 
 
Задачи курсового проекта: 
 1. Провести предпроектное обследование и бизнес -анализ предметной 
области.  
2. Разработать модель требований к программному обеспечению. 
3. Спроектировать архитект уру программной системы на основе 
паттерна PCMEF. 
4. Реализовать ядро системы с соблюдением принципов SOLID и 
слоевой архитектуры  
5. Обеспечить качество кода через тестирование и рефакторинг. 
6. Разработать пользовательский интерфейс в соответствии с выбранной 
траекторией. 
7. Обеспечить безопасность и управление транзакциями. 
8. Предоставить API для интеграции с внешними системами (для 
соответствующих траекторий)  
9. Выполнить развертывание системы  
10. Подготовить полный комплект проектной документации

 1.3. ВЫБОР ТРАЕКТОРИИ ПРОЕКТА 
 1.3.1. Общая концепция: PCMEF как единая архитектурная основа 
В основе всех траекторий курсового проектирования лежит 
архитектурный паттерн PCMEF (Presentation -Control-Mediator-Entity-
Foundation). Этот паттерн является концептуальным каркасом, который: 
- Обеспечивает четкое разделение ответственности между 
компонентами; 
- Гарантирует направленность зависимостей сверху вниз; 
- Позволяет тестировать каждый слой изолированно; 
- Адаптируется под различные среды выполнения. 
 
Структура PCMEF: 
Слой Английское 
название 
Русское 
название 
Ответственность 
P Presentation Представление Интерфейс по льзователя, 
отображение данных 
C Control Управление Обработка запросов, 
координация, валидация 
M Mediator Посредник Бизнес-логика, транзакции, 
правила 

--- PAGE 7 ---
E Entity Сущность Бизнес-объекты, состояние, 
методы предметной области 
F Foundation Фундамент Доступ к данным, 
инфраструктура, репозитории 
 
Правило зависимостей:  Зависимости направлены строго вниз . 
Вышележащие слои могут  зависеть от нижележащих, но не наоборот. Это 
обеспечивает стабильность архитектуры.

class EventViewModel : ViewModel() { 
    private val _state = MutableStateFlow<EventState>(EventState.Loading) 
    val state: StateFlow<EventState> = _state.asStateFlow() 
     
    private val repository: EventRepository 
     
    fun loadEvents() { 
        viewModelScope.launch { 
            _state.value = EventState.Loading 
            try { 
                val events = repository.getEvents() 
                _state.value = EventState.Success(events) 
            } catch (e: Exception) { 
                _state.value = EventState.Error(e.message ?: "Unknown error") 
            } 
        } 
    } 
}

class EventBloc extends Bloc<EventEvent, EventState> { 
  final EventRepository repository; 
   
  EventBloc(this.repository) : super(EventInitial()) { 
    on<LoadEvents>((event, emit) async { 
      emit(EventLoading()); 
      try { 
        final events = await repository.getEvents(); 
        emit(EventLoaded(events)); 
      } catch (e) { 
        emit(EventError(e.toString())); 
      } 
    }); 
  } 
}
